# 项目开发规则

本项目的开发者是现代软件开发初学者。请在保证工程质量的同时，让每次修改都容易理解、运行、验证和撤销。

## 基本技术约束

* 前端使用 Vue 3、TypeScript、Quasar 和 Vite。
* 状态管理使用 Pinia。
* 图表使用 Apache ECharts。
* 后端使用 Supabase，包括 PostgreSQL、Auth、RLS、Realtime 和 Storage。
* 第一阶段为响应式 Web 应用。
* 不在当前阶段引入 Flutter、Python 服务、Tauri、完整离线同步或微服务。
* 不得把 Supabase service role 密钥放入客户端。
* 所有家庭业务表必须考虑 family_id 和 RLS 权限隔离。

## 每次任务的工作方式

1. 修改前先阅读相关文件和项目文档。
2. 先说明准备修改什么、为什么修改、涉及哪些文件。
3. 每次只完成一个明确且可以独立验证的小任务。
4. 不要顺便重构无关代码。
5. 不要一次生成整个模块或大量占位代码。
6. 优先使用简单、稳定、主流的实现，不做过度设计。
7. 修改后必须运行现有的类型检查、代码检查和测试。
8. 如果没有对应测试，请为关键业务逻辑补充最小测试。
9. 测试失败时先分析原因，不得删除测试来掩盖错误。
10. 完成后说明如何由初学者手动验证结果。

## 代码可读性

* 使用明确完整的英文变量名、函数名和数据库字段名。
* 页面显示文字使用中文。
* 对权限判断、日期计算、统计规则和数据同步等关键逻辑添加中文注释。
* 不要给显而易见的代码逐行添加无意义注释。
* 函数保持简短，复杂逻辑拆成有明确名称的小函数。
* 禁止使用无法解释来源的数字和字符串，使用常量或枚举。
* 不要使用 any 绕过 TypeScript 类型检查，除非说明原因。
* 错误信息应能让普通用户理解。

## 数据库修改

* 数据库结构变更必须通过迁移文件完成。
* 不得直接破坏已有数据。
* 新增表时说明主键、外键、唯一约束、索引和 RLS。
* 涉及删除时优先考虑软删除。
* 日期、时间和时区规则必须明确。
* 权限策略必须分别验证家长、孩子和非家庭成员。

## 完成任务时的说明

请使用以下格式汇报：

1. 本次完成内容；
2. 修改文件；
3. 关键代码解释；
4. 已执行的检查和测试；
5. 手动验证步骤；
6. 仍存在的限制；
7. 建议的下一个最小任务。

如果我的要求可能造成架构问题，请先解释风险并给出更简单的替代方案，但不要未经说明擅自扩大任务范围。
