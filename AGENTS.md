# AGENTS.md

本文件是Codex及其他编码代理在本仓库中工作时必须遵守的长期规则。

项目开发者是现代软件开发初学者。所有修改必须容易理解、运行、验证、撤销，并适合通过GitHub Desktop在多台设备间继续开发。

完整产品需求、数据模型和阶段计划不在本文件重复。

## 1. 项目文档读取顺序

开始任何任务前，依次检查：

1. `AGENTS.md`
2. `README.md`
3. `docs/PROJECT_STATUS.md`
4. `docs/NEXT_TASK.md`
5. 与任务有关的文档：

   * `docs/PRODUCT.md`
   * `docs/DATA_MODEL.md`
   * `docs/DECISIONS.md`
   * `docs/ROADMAP.md`
   * `docs/DEVELOPMENT_WORKFLOW.md`
6. 最近的`docs/CHANGELOG.md`内容
7. 实际代码、测试和数据库迁移

如果文档与实际实现冲突：

* 代码和数据库迁移代表当前实现事实；
* Git历史代表实际变更事实；
* 先报告冲突；
* 不得根据过时文档覆盖有效代码；
* 在当前任务范围内同步修正文档。

## 3. 开始任务前的Git检查

修改任何文件前必须检查：

* 当前分支；
* `git status`；
* 是否有未提交修改；
* 是否有合并冲突；
* 最近提交；
* 当前任务是否与`docs/NEXT_TASK.md`一致；
* 本地分支是否可能落后于远端。

不得假设当前分支一定是`main`。

如果本地可能落后于远端，应提醒用户在GitHub Desktop执行：

1. `Fetch origin`
2. `Pull origin`

同步完成前不要修改代码。

如果存在来源不明的未提交修改，不得覆盖、删除或擅自保存到Stash。

## 4. 任务范围

每次只实现一个可以独立验证的小任务。

修改前先说明：

* 本次目标；
* 计划修改的文件；
* 明确不修改的内容；
* 验收方式。

禁止：

* 顺便重构无关代码；
* 一次实现整个模块；
* 创建大量未来占位代码；
* 未经说明改变架构；
* 为简单功能创建无意义抽象；
* 删除测试来掩盖错误；
* 使用`any`逃避类型检查；
* 为显而易见代码添加大量逐行注释。

关键权限、日期、统计、同步和业务规则需要添加简洁的中文解释。

## 5. 前端职责和依赖方向

遵守以下方向：

页面和组件
→ Composable
→ Repository
→ Service / Supabase Client

职责：

* Page负责页面组合；
* Component负责可复用显示和交互；
* Composable负责页面流程、加载和错误状态；
* Repository负责业务实体的数据读取和写入；
* Service负责Supabase客户端、Storage和Realtime等基础设施；
* Pinia只保存登录会话、当前家庭、当前孩子及少量全局状态；
* Domain保存日期、年级、统计等纯业务函数。

不得：

* 在Vue页面中直接访问Supabase业务表；
* 在页面组件中拼接SQL；
* 把复杂统计或权限逻辑放入模板；
* 为每个页面机械创建ViewModel；
* 长期手写一套与数据库不一致的重复类型。

数据库类型应尽量从Supabase schema生成。

## 6. 数据库和RLS安全

数据库结构修改必须通过版本化迁移文件完成。

不得：

* 只在Supabase控制台手工改表而不补迁移；
* 把service role、secret key或数据库密码写入前端；
* 依靠隐藏按钮实现权限；
* 未启用RLS就暴露家庭业务表；
* 仅根据客户端提交的`family_id`授权；
* 未完成越权测试就宣布权限功能完成。

RLS修改至少验证：

1. 同一家庭成员的合法访问；
2. 孩子只能访问自己的数据；
3. 家庭A不能读取家庭B；
4. 非家庭成员不能访问；
5. 已退出成员失去访问权；
6. 手工构造请求不能越权；
7. 孩子不能访问家长工作和费用数据。

使用`SECURITY DEFINER`函数时必须：

* 说明使用原因；
* 固定安全的`search_path`；
* 使用完整schema表名；
* 限制执行权限；
* 优先从`auth.uid()`取得当前用户；
* 编写正向和反向权限测试。

## 7. 密钥和公开仓库安全

本仓库可能公开访问，任何真实密钥和家庭数据都不得进入Git。

必须：

* 提交`.env.example`；
* 忽略`.env`、`.env.local`及其他真实环境文件；
* 不提交`node_modules`；
* 不提交数据库密码；
* 不提交Supabase service role key；
* 不提交孩子资料、试卷、证书、真实导出数据或测试照片；
* 不提交本机绝对路径；
* 不提交日志、临时导出和调试文件。

前端只允许使用Supabase项目URL和允许公开的publishable/anon key，并且必须以正确RLS为安全前提。

发现疑似密钥时立即停止任务并报告。

## 8. 测试和检查

修改后运行适用检查。

项目初始化完成后，以`package.json`中的实际脚本为准，通常包括：

* 类型检查；
* Lint；
* 单元测试；
* 组件测试；
* 构建检查。

优先测试：

* 3月1日和9月1日学期边界；
* 日期、时区和跨午夜时长；
* 得分率；
* 学科统计和任务量分组；
* 家庭和孩子权限；
* RLS隔离；
* Realtime断线、重连和重复事件。

如果检查无法运行，必须说明：

* 哪项检查未运行；
* 原因；
* 用户如何手动运行；
* 当前仍有什么风险。

## 9. GitHub Desktop和多设备开发

`main`应尽量保持可启动、可检查。

分支规则：

* 小型纯文档修改可直接在`main`完成；
* 新功能、数据库迁移、RLS和高风险修改使用短期任务分支；
* 不创建按电脑命名的长期分支；
* 同一任务跨设备时继续使用同一个任务分支；
* 同一时刻只在一台设备主动修改同一分支。

推荐分支名称：

* `feat/...`
* `fix/...`
* `docs/...`
* `test/...`
* `chore/...`

Codex默认不得执行：

* commit；
* push；
* merge；
* rebase；
* force push；
* reset --hard；
* git clean；
* commit --amend；
* 删除分支；
* 丢弃用户修改；
* 改写Git历史。

只有用户在当前任务中明确授权后才可执行Git写操作。

Codex可以读取：

* `git status`
* `git diff`
* `git log`
* 当前分支和远端信息

不得把Stash当作跨设备交接方式，因为Stash只保存在当前设备。

## 10. 多设备交接文档

完成有效开发后，根据实际影响更新：

### `docs/PROJECT_STATUS.md`

记录：

* 当前里程碑；
* 已稳定完成内容；
* 正在进行内容；
* 数据库、RLS、Realtime和Storage状态；
* 最近通过的检查；
* 已知问题和阻塞项。

### `docs/NEXT_TASK.md`

任何时候只保留一个主要下一任务。

必须包含：

* 状态；
* 任务名称；
* 所属里程碑；
* 推荐分支；
* 目标；
* 本次范围；
* 明确不做；
* 验收标准；
* 自动检查；
* 手动验证；
* 当前停止位置；
* 下一步第一项操作。

完成旧任务后必须将其替换为新的下一任务。

### `docs/CHANGELOG.md`

只记录：

* 用户可感知的新功能；
* 重要架构变化；
* 数据库迁移；
* 权限和安全修复；
* 重要兼容性修复；
* 开发流程的重要调整。

不要机械复制Git提交记录。

其他文档：

* 产品需求变化更新`PRODUCT.md`；
* 数据模型变化更新`DATA_MODEL.md`；
* 重大技术决策更新`DECISIONS.md`；
* 阶段变化更新`ROADMAP.md`；
* GitHub Desktop流程变化更新`DEVELOPMENT_WORKFLOW.md`。

## 11. 切换设备前的交接

结束工作或切换设备前必须：

1. 停在明确检查点；
2. 运行适用检查；
3. 更新`PROJECT_STATUS.md`；
4. 更新`NEXT_TASK.md`；
5. 必要时更新`CHANGELOG.md`；
6. 检查是否包含密钥和本地文件；
7. 检查GitHub Desktop的Changes列表；
8. 提供建议提交信息；
9. 指导用户Commit；
10. 指导用户Push origin；
11. 提醒确认工作区没有遗留修改。

任务未完成但必须切换设备时：

* 只能在任务分支建立明确的WIP检查点；
* `NEXT_TASK.md`必须记录未完成部分和当前失败项；
* 不得把破损WIP合并到`main`。

新设备开始时：

1. 在GitHub Desktop Fetch；
2. Pull当前任务分支；
3. 阅读`PROJECT_STATUS.md`；
4. 阅读`NEXT_TASK.md`；
5. 安装依赖；
6. 创建本机环境变量；
7. 运行基础检查；
8. 从“下一步第一项操作”继续。

## 12. 完成汇报格式

每次任务结束后按以下格式汇报：

1. 本次完成内容
2. 修改文件
3. 关键设计解释
4. 已运行检查和结果
5. 初学者手动验证步骤
6. 文档更新情况
7. GitHub Desktop提交建议
8. 已知限制
9. 下一最小任务

任务只有同时满足以下条件才算完成：

* 符合当前任务范围；
* 类型检查和适用测试通过；
* 没有已知越权风险；
* 没有提交密钥和真实家庭数据；
* 文档和实现一致；
* 用户可以按步骤手动验证；
* 已准备好跨设备接续信息。

如果用户的要求可能造成架构、安全或维护问题，先解释风险并给出更简单的替代方案，不得未经说明擅自扩大任务范围。
